# Project Principles

## Open Client, Honest Boundary

The client can be open source without making the entire official POKROV service
open source.

Documentation should make that boundary easy to understand.

## Users First

The client should make connection, recovery, support, and subscription status
clear to normal users. Advanced runtime complexity should not leak into the
first-layer user experience.

## Evidence-Based Releases

Release notes should describe what is proven, not what is hoped for.

Avoid unsupported claims about store availability, stable releases, trusted
Windows signing, raw Android audit evidence, RU-origin readiness, or production
WARP readiness.

## Reproducible Public Source

Public source should build from a clean clone with documented tools and public
configuration examples.

## Forks Are Welcome, Impersonation Is Not

Forks are part of open source. Misleading users into thinking a fork is an
official POKROV build is not.
